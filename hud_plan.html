HUD Integration Plan — SO101 Gold-Bar Pickup

Consolidated plan for wiring our SO101 gold-bar → jar task into HUD v6 (robot capability), plus an honest read on where HUD actually earns its keep vs. where LeRobot already does the job. This is the HUD-specific companion to PLAN.md.

0. TL;DR — the one decision that matters

There are two recording paths, and they are not equivalent:

Sim loop (Isaac Lab) Real arm
Who records the dataset HUD (agent-side, from the contract) LeRobot (lerobot-record, owns the hardware)
Who computes reward HUD bridge result() (reads sim state) our CV/VLM scorer (no ground truth)
Does HUD do real work? Yes — load-bearing Mostly the demo/trace/scoring layer

Conclusion up front (answers the research question): - In sim, HUD genuinely creates the dataset. Isaac’s dep stack can’t even import lerobot; HUD records the rollout agent-side (on our laptop) straight into a LeRobot v3 dataset from the contract, and gives us the reward via the sim bridge. That is real, non-trivial work we’d otherwise hand-roll. - On the real arm, lerobot-record --policy.path=... already runs the policy AND records a v3 dataset in one command — it owns the SO101 bus, cameras, episode segmentation, resets, re-record keys, and Hub push. Using HUD to record the real arm means writing a hardware RobotBridge that duplicates LeRobot. So for the real loop, HUD’s honest value is the trace viewer (curation + demo) and the scored-eval presentation — i.e. largely “run it in post and trace the gold bar.” Legitimate, but thin.

Therefore: make the sim rollout loop the place HUD is load-bearing; on the real arm, record with LeRobot and use HUD for tracing + scored eval.

1. Architecture (chosen path)

  • Embodiment: SO101, 6-DoF absolute joint positions (5 arm + gripper). Same action space across real / sim / deploy so an ACT checkpoint maps 1:1.
  • Sim: our GitHub SO101 Isaac Lab env + colleague’s OpenUSD scene + jar + graspable gold bar.
  • HUD wrapper: the robolab two-process pattern from robot-template (inventory/envs/robolab/). Isaac owns the process main thread, so the sim runs in its own process and the lightweight HUD env connects to it.
  • Policy: ACT, wrapped as a HUD RobotAgent via LeRobotModel + LeRobotAdapter (ACT is a stock LeRobot policy).
                  ┌─ Isaac process (isaaclab python) ────────────────────┐
                  │  sim_process.py                                       │
   HUD env.py     │   SO101Bridge(reset/step/get_observation, result)     │
 (lightweight) ──▶│   RobotEndpoint(bridge).serve(127.0.0.1, 9100)        │
 RobotEndpoint    │   MainThreadSimRunner  +  USD scene + jar + gold bar  │
   .remote(...)   └───────────────────────────────────────────────────────┘
        │
        ▼ openpi/0 WebSocket (observe → act), data plane straight to bridge
   RobotAgent(ACT)  ──  LeRobotModel + LeRobotAdapter   (laptop, has lerobot)
        │
        └─ agent.save=True ▶ LeRobot v3 dataset (RECORD_DIR) ▶ retrain

Two planes (per the docs): control plane (start/reset/result/stop) is JSON-RPC between env and sim process; data plane (the agent’s observe→act loop) tunnels straight to the bridge WebSocket. The agent never knows the sim is remote.

2. What we actually have to write

Copy robot-template/inventory/envs/robolab/ and inventory/agents/ and edit. Concretely:

2.1 so101_joint_pos.json — the contract (NEW, ~40 lines)

The single shared artifact. Drives both space-wiring and the recorded dataset schema.

{
  "robot_type": "so101",
  "control_rate": 30,
  "features": {
    "observation/overhead_image": { "role": "observation", "type": "rgb",
      "names": ["height", "width", "channel"], "shape": [480, 640, 3] },
    "observation/wrist_image":     { "role": "observation", "type": "rgb",
      "names": ["height", "width", "channel"], "shape": [480, 640, 3] },
    "observation/state": { "role": "observation", "type": "joint_pos",
      "names": ["shoulder_pan", "shoulder_lift", "elbow_flex", "wrist_flex", "wrist_roll", "gripper"],
      "shape": [6] },
    "action": { "role": "action", "type": "joint_pos",
      "names": ["shoulder_pan", "shoulder_lift", "elbow_flex", "wrist_flex", "wrist_roll", "gripper"],
      "shape": [6] }
  }
}
  • control_rate must equal the fps the real ACT data was recorded at — the agent uses it as the recorded-dataset fps and trace-viewer playback speed.
  • type: "joint_pos" on state/action is descriptive; type: "rgb" on images is load-bearing (it’s how the adapter spots a camera vs. the state vector).
  • Feature keys are verbatim what the bridge returns and what the recorder names the dataset columns: observation/overhead_image → observation.images.overhead_image, observation/state → observation.state, action → action.

2.2 so101_sim_bridge.py — the bridge (the bulk of the work)

Three methods (HUD owns the serve loop, codec, control-rate stepping): - reset(task_name, seed) → load USD scene + SO101 + jar + gold bar, settle, return prompt (“pick up the gold bar and put it in the jar”). - step(action) → write the 6 joint targets to the SO101 articulation, advance the sim, update success / terminated / score from the reward detector (§3). - get_observation() → {"observation/overhead_image": rgb, "observation/wrist_image": rgb, "observation/state": joints6}, terminated. Images HWC uint8; state 1-D float32. - result() → {"score", "success", "total_reward"}.

Model after inventory/envs/robolab/robolab_sim_bridge.py, but swap the DROID env/action/obs config for our SO101 Isaac env’s API. This is the one place I need the GitHub repo’s env interface to fill in.

2.3 sim_process.py + env.py + config.py (mostly copy)

  • sim_process.py: boot Omniverse, build the bridge with MainThreadSimRunner, RobotEndpoint(bridge).serve(...). Copy from robolab almost verbatim.
  • env.py: RobotEndpoint.remote(host, port), publish the capability with our contract, one pick_and_place template. Copy from robolab.
  • config.py: point at so101_joint_pos.json; set device, endpoint host/port.

2.4 so101_act_agent.py — the policy agent (~15 lines)

Mirror inventory/agents/smolvla_libero.py, swap in ACT:

from lerobot.policies.act.modeling_act import ACTPolicy
from hud.agents.robot import RobotAgent, LeRobotModel, LeRobotAdapter

class SO101ACTAgent(RobotAgent):
    def __init__(self, checkpoint, device="cuda"):
        policy = ACTPolicy.from_pretrained(checkpoint).to(device).eval()
        pre, post = make_pre_post_processors(policy.config, checkpoint,
            preprocessor_overrides={"device_processor": {"device": device}})
        self.model = LeRobotModel(policy, pre, post)   # pass an Ensembler (ACT is chunked)
        self.adapter = LeRobotAdapter(model_image_keys=list(policy.config.image_features))

ACT predicts chunks → reduce to one action/step with an Ensembler (temporal ensembling), matching how ACT runs upstream.

2.5 Reward / success detector (NEW — build now, needs no model) — see §3.

Net new code: 1 contract + 1 reward module + the bridge body (everything else is copy-and-rename). The bridge body is the only real unknown, gated on the SO101 Isaac env API.

3. Reward / success detector (the rollout judge)

In step(), read object + jar poses from the Isaac scene: - success = gold-bar xy inside jar radius AND z below the rim AND a was_lifted flag set earlier in the episode (so “shoved off the table” ≠ success). A horizon timeout is not a success. - partial credit (shaping, for top-K / debugging): score = 0.5 * lift_progress + 0.5 * (1 - clamp(horizontal_dist_to_jar)).

Mirror worldsim’s _lift_progress / _reward shape and expose via result(), exactly like RobolabSimBridge.score. Validate it with a scripted pick (approach → close → lift → place) before involving the policy — that also de-risks the physics (§6).

4. Recording — exactly how HUD does it (grounded in the v6 docs)

Docs §“Recording datasets”: Set agent.save = True to record every (observation, executed action) tick into a LeRobot v3 dataset. Recording is agent-side — it consumes the observations the agent already receives and the actions it produces, so it runs in your process, not the environment container. Only your machine needs pip install 'lerobot[dataset]'.

  • The contract drives the schema, no extra wiring: image features → observation.images.<name> (encoded to per-episode video), the lone state vector → observation.state, the action → action, the prompt rides along as each frame’s task.
  • Env vars (set on the env): RECORD_DIR (dataset root, default ./data), HF_REPO (+ HF_TOKEN to push), HF_PRIVATE. (Note: the robot-template README calls these HUD_RECORD_DIR / HUD_HF_REPO — verify which the installed hud-python build expects; the docs say RECORD_DIR/HF_REPO.)
  • One dataset spans the whole run, finalized at process exit. → For filtered-BC we get one v3 dataset of N episodes; we then drop episodes by index using their per-episode scores (v3 datasets are episode-aware: meta/episodes/). Or run scoring in per-batch processes.

Why this is the load-bearing bit in sim: the reward comes free from the bridge result(), and the dataset comes free agent-side despite Isaac not having lerobot installed. That combination is exactly what makes the sim self-improvement loop cheap.

5. The self-improvement loop (filtered BC, sim)

Online RL on the policy head is out of scope (research-grade). We do reward-filtered behavior cloning, and we bootstrap so the filter has signal:

  1. Sim teleop bootstrap (~15–20 demos): drive the SO101 sim with the real leader → recorded via HUD agent-side (or LeRobot). Anchors the sim visual domain.
  2. Retrain ACT co-trained = real demos + sim-teleop demos. Now it works in sim.
  3. Loop ×2–3: roll out (Gizmo/USD domain randomization) → HUD scores each episode → keep successes / top-K → append → retrain (always include real data).
  4. Stop early if real eval regresses (catastrophic-forgetting guard).

Throughput note: robolab is single-env (num_envs=1, ~2 min Omniverse boot) — fine for the scored eval + demo, slow for mass rollouts. If the SO101 Isaac env supports vectorized num_envs=N, generate bulk rollout data with a separate native vectorized script (policy + reward detector → dump v3), and keep the HUD bridge for the polished scored runs.

6. Critical gotchas (in priority order)

  1. Joint-convention alignment (the sim2real killer). Real follower joint order/sign/zero-offset/units (LeRobot+Feetech) must match the Isaac SO101 exactly, or the ACT checkpoint maps to garbage. Verify: command one joint vector to real and sim, confirm identical pose. Build a calibration map if needed.
  2. Graspability > physics perfection. Only object mass, finger/object friction, and gripper force/gain matter for a pick. Tune those via the scripted pick (§3); don’t hand-tune the whole URDF. (This is where the “unsure if physics values are good” risk lives.)
  3. Camera pose/FOV/lighting match real overhead rig ↔︎ sim camera.
  4. Env var name drift (RECORD_DIR vs HUD_RECORD_DIR) — verify against installed build.
  5. Filtered BC needs >0 sim success — hence the teleop bootstrap (§5).

7. While the model trains — ordered checklist

  1. Lock the joint convention map (real ↔︎ Isaac SO101).
  2. Get the GitHub SO101 Isaac env to reset/step standalone; load the USD scene + jar + gold bar.
  3. Scripted pick → validate graspability (tune mass/friction/gripper only).
  4. Build the reward/success detector (§3).
  5. Write so101_joint_pos.json (§2.1) — can do now, no deps.
  6. Wrap in the robolab bridge (§2.2–2.4); prove a NoopAgent rollout streams to the HUD trace viewer and agent.save=True writes a valid v3 dataset (pip install 'lerobot[dataset]').
  7. Match the overhead camera to the real rig.
  8. Smoke-test sim teleop (real leader → sim).
  9. Last / polish: Gizmo/USD background clutter + lighting randomization per reset().

8. Real-arm track (where HUD is just the demo/trace layer)

For the real loop, don’t reinvent the hardware in a HUD bridge — use LeRobot:

# autonomous rollouts + dataset in one command (native v3 recording)
lerobot-record \
  --robot.type=so101_follower --robot.port=/dev/ttyACM0 --robot.id=follower \
  --robot.cameras='{ overhead: {...}, wrist: {...} }' \
  --dataset.repo_id=$HF_USER/so101_goldbar_rollouts \
  --dataset.single_task="Pick up the gold bar and put it in the jar" \
  --dataset.num_episodes=200 \
  --policy.path=outputs/train/act_goldbar/checkpoints/last/pretrained_model

Then: CV scorer (gold-bar HSV centroid inside the fixed jar ROI + was_lifted gate; VLM keyframe judge as audit) labels each episode → filter / reward-weight → retrain. HUD’s role here: trace viewer for curation + the demo, and the scored-eval presentation. Prefer HG-DAgger (human-gated teleop correction) over pure success-filtering — at ~15% success, mining autonomous successes is low-yield (≈200 rollouts with manual resets for ~30 successes); interventions target the policy’s weak states and are far more sample-efficient.
